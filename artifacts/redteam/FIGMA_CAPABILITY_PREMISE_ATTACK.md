# Redteam I — Attack the Portfolio Figma Premise

## Claim under attack

A Figma capability can improve every active repository by giving agents a repeatable build-and-implementation path.

## Failure modes

1. A single generic skill ignores framework and product differences.
2. Figma becomes a second source of product or operational truth.
3. Agents create polished screens without implementing them.
4. Code is forced to match a mockup that invented APIs, schema, products, or authority.
5. Private data enters Figma cloud files, screenshots, libraries, or Code Connect examples.
6. Public and private commerce libraries become cross-connected.
7. Code Connect is claimed without published components, plan eligibility, exact node IDs, or verified props.
8. Design-system work creates new tokens/components while compatible local or subscribed assets already exist.
9. A screenshot or prototype is mislabeled as exact-head, native-device, checkout, migration, or deployment proof.
10. Parallel design and code agents overwrite one another or implement different states.

## Premise correction

The useful capability is not "Figma everywhere." It is a shared protocol plus a repository-owned profile that binds Figma to the actual framework, data boundary, authority model, implementation paths, and proof stack.

## Stop conditions

Stop the design lane when the repo profile is missing, the source-of-truth code cannot be identified, prohibited data is required, the framework is unresolved, the requested change needs unapproved authority, or the design cannot be verified against an implementation target.
