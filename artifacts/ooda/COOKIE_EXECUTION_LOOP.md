# OODA — Cookie Contract Execution Loop

## Observe

- Founder Control Room had an emerging browser session but incomplete CSRF and cache boundaries.
- Se’kret Bip already used SecureStore on native and did not need browser-cookie imitation.
- Untold Stories already had a Hydrogen session cookie but lacked production Secure naming and no-store commits.
- JBH Private already had Cloudflare Access assertion validation; its local password screen was not a production route.
- L99, Chief AI, and the public hair storefront had no justified first-party session cookie.

## Orient

Cookies are transport-bound session mechanisms, not a universal persistence layer. Portfolio consistency should live in machine-readable contracts and verifiers while implementations remain runtime-specific.

## Decide

- Harden the two first-party server cookies.
- Declare the Cloudflare Access provider cookie without duplicating it.
- Make the remaining repositories cookie-free by default.
- Document Stripe and Shopify provider boundaries.
- Add exact source scans and no-store/CSRF invariants.
- Integrate verifiers into Founder Control Room exact-head CI and GitHub status mirroring.

## Act

1. Land repo-local manifests and verifiers in isolated draft branches.
2. Execute each verifier against its exact branch head.
3. Repair false positives or undeclared writers.
4. Add Founder Control Room command registry entries.
5. Produce one portfolio attestation bound to every repository SHA.
6. Mirror statuses to GitHub.
7. Merge only after complete proof and explicit founder approval.

## Re-observe

A future cookie, SDK, analytics script, auth change, or provider migration must update the manifest first. Any undeclared writer or weakened sensitive-cookie boundary becomes drift and blocks promotion.
