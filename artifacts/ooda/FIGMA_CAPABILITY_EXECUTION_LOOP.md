# OODA — Portfolio Figma Execution Loop

## Observe

Read the local repository profile, agent contract, current branch, implementation framework, component paths, design files, subscribed libraries, variables, components, tests, and active proof blockers.

## Orient

Classify the task:

- design-only exploration;
- production design-system work;
- screen/flow specification;
- design-to-code implementation;
- Code Connect mapping;
- visual QA against an implementation.

Map data classification, authority, user, state machine, responsive targets, accessibility, implementation owner, and proof commands.

## Decide

Choose the smallest coherent path:

- reuse local component;
- import compatible subscribed component;
- wrap a compatible library component;
- create a new local component;
- capture an existing web page;
- implement directly from an approved design;
- stop because the source of truth or authority is missing.

## Act

Build foundations, components, and screens in order. Implement in the verified repo framework. Run the second Redteam. Execute the local proof stack. Record exact Figma and repository identifiers.

## Loop triggers

Re-enter Observe when code, Figma, product data, auth, schema, provider, catalog, deployment, or test evidence changes; when a design review changes scope; when implementation reveals a missing state; or when exact-head proof invalidates local assumptions.
