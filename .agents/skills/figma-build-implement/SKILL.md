# Founder Control Room Figma Build + Implementation

Load this skill for every Figma, dashboard design, design-system, prototype, design-to-code, visual QA, or Code Connect task in this repository.

## Required tool skills

- Load `figma-use` before every Figma write.
- Load `figma-generate-library` for tokens, variables, reusable components, variants, themes, or a component library.
- Use `figma-generate-design` only to capture a running web view into an existing file, then rebuild editable structure with `figma-use`.
- Use `figma-code-connect` only after published components, plan eligibility, a node-specific URL, and a verified code component are available.

## Repository profile

- Runtime: TypeScript, Express, Supabase, Cloudflare Worker paths, and repository-specific web/control surfaces.
- Primary design targets: founder command center, portfolio evidence, mission review, approval gates, verification runs, release state, economic intelligence, opportunity scoring, and operational status.
- Source of truth: current repository types, routes, schemas, evidence contracts, authorization middleware, and exact runtime behavior. Inspect the current frontend implementation instead of assuming React or another framework.
- Figma is a founder review and specification surface. It must not become a second mission ledger, approval authority, provider state source, or deployment controller.

## Mandatory operating sequence

1. Run the repository 5W1H and founder reasoning stack.
2. Redteam the premise: does the requested surface reduce founder uncertainty, or merely decorate unverified state?
3. Inspect current APIs, schema, auth, evidence provenance, project boundaries, active branches, and existing Figma libraries.
4. Lock the exact user, route, state machine, visibility, data classification, implementation path, and proof commands.
5. Build variables and reusable components before screens. Reuse subscribed design-system assets when their component API and token model fit.
6. Design explicit states for verified, observing, degraded, blocked, stale, unauthorized, empty, loading, failure, and rollback-ready conditions.
7. Implement only against verified repository contracts. Never invent an API or database column to satisfy a mockup.
8. Redteam the selected implementation: false-green state, stale evidence, wrong SHA, approval carry-forward, secret leakage, cross-project data, and inaccessible failure paths.
9. Verify with the repository's typecheck, lint, unit/contract tests, Playwright, and exact-head evidence policy.
10. Record Figma URLs/node IDs, code paths, evidence, known drift, rollback, and the next founder gate.

## Data and authority boundary

- Use synthetic or sanitized operational data in Figma.
- Never include service-role keys, provider tokens, raw webhook payloads, raw product content, teen/family content, customer data, vendor data, or unpublished founder strategy.
- A button in Figma does not authorize merge, deployment, migration, spending, rollback, secret access, or external communication.
- No approval carries forward. Every material action remains separately gated.
- Public-safe designs must expose only sanitized contract/status information.

## Code Connect contract

Code Connect remains conditional. Map only published reusable components with exact node IDs and verified code props. Exhaustively map variants and preserve framework neutrality where the repository has not selected a frontend framework. Do not create ornamental mappings for one-off dashboard frames.

## Definition of done

The design, implementation status, state coverage, accessibility, provenance, security review, exact tests, unresolved differences, rollback, and next approval gate are all documented. A polished dashboard with simulated data is not operational proof.
