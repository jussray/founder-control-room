# Founder Control Room Agent Instructions

## Required repository skills

Read these before nontrivial work:

- `.agents/skills/founder-control-room-operator/SKILL.md` for 5W1H, repository identity, authority, proof, and rollback;
- `.agents/skills/sales/SKILL.md` for positioning, offers, qualification, conversion, retention, partnerships, and revenue operations;
- `.agents/skills/devil/SKILL.md` for the premise attack and selected-plan attack before material execution.

Use 5W1H as active reasoning: establish who, what, where, when, why, and how; inspect missing answers; and ask only when an unknown materially changes the safe action or authority.

Read [`GLOBAL_AI.md`](./GLOBAL_AI.md) before changing code, configuration, schemas, providers, or documentation.

Use the exact founder stack:

```text
/garyvee lindymode redteam l99 redteam ooda
```

For commercial work, add the paired extension:

```text
/sales /devil
```

`/sales` constructs the strongest truthful exchange. `/devil` attacks both the premise and the selected plan. Neither authorizes outreach, pricing, discounts, spending, publication, checkout, merge, deployment, or database mutation.

## Required loop

1. Inspect the repository, branch, provider state, auth boundaries, migrations, tests, and relevant runtime evidence.
2. Complete 5W1H and identify the next authority gate.
3. Attack the premise before designing a solution.
4. Apply the Lindy screen and map authority, provenance, project boundaries, event history, sales state, and rollback through L99.
5. Attack the selected plan before implementation.
6. Make the smallest coherent change and verify it.
7. Re-observe through OODA and report proof plus the next founder approval gate.

## Non-negotiable boundaries

- Preserve the `RepositoryProvider` abstraction.
- Keep Control Room Supabase, credentials, and data separate from Se’kret Bip.
- Never copy raw teen, journal, voice, media, health, parent-visibility, customer-secret, or other sensitive user data into operational storage, prompts, sales analysis, or outreach.
- Preserve founder allowlist authorization, audit events, project isolation, and separate approval gates.
- Do not expose provider tokens or service-role keys.
- Do not invent demand, scarcity, customer statements, eligibility, savings, performance, inventory, or authority.
- Do not merge, deploy, roll back, alter auth/RLS, contact external parties, change commercial terms, spend funds, or perform destructive writes without explicit founder approval for that exact action.

## Evidence report

List files changed, behavior changed, tests run, failures or skips, security impact, provider impact, commercial assumptions, disqualifiers, proof, rollback, unresolved risk, and next gate.
